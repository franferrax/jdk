Compatibility Risk: low

Compatibility Risk Description:
===============================

If a PKCS #11 token supports CKM_AES_CTS with a different ciphertext ordering variant than SunPKCS11's default (CS1), the new "cipherTextStealingVariant" configuration parameter has to be set to a value of CS2 or CS3. This issue may cause some tokens not to work out of the box. While this interoperability issue exceeds OpenJDK (see section "Problem" of this CSR), we have introduced a configuration parameter to support different ordering variants, made public documentation available herein and provided the test TestCipherTextStealingMultipart to cross-check encryption and decryption results against SunJCE's CTS implementation.

In addition, an application that is using any of the "AES/CTS/NoPadding" algorithms already (directly or indirectly through Kerberos) while listing SunPKCS11 with higher precedence than SunJCE or any 3rd party security provider, may experience compatibility issues. If the SunPKCS11 underlying token supports the CKM_AES_CTS mechanism and uses an ordering variant different from CS1, the AES CBC-CTS operation will be done in the token and the result will be incorrect until the "cipherTextStealingVariant" configuration is adjusted or SunPKCS11 is not listed with higher precedence.

We deem the risk for this proposal to be low, documented and addressable by means of configuration adjustments.

-------------------------------

Summary
-------

Update the _SunPKCS11_ security provider to support the AES CBC with Ciphertext Stealing (CBC-CTS) `Cipher` algorithm for keys of 128, 192 and 256 bits. The `CKM_AES_CTS` mechanism is required in the underlying PKCS #11 token for this feature to be available.

Problem
-------

The _SunPKCS11_ security provider does not support the AES CBC-CTS algorithm at the moment. However, the PKCS #11 standard defines the `CKM_AES_CTS` native mechanism and implementations such as the [NSS Software Token](https://github.com/nss-dev/nss/blob/NSS_3_90_RTM/lib/freebl/cts.c) are available. This gap prevents OpenJDK from benefiting of the algorithm for `Cipher` operations, even when supported by the underlying PKCS #11 token.

In addition, the AES CBC-CTS algorithm can be used for Kerberos authentication (e.g. with CTS-based encryption types such as `aes256-cts-hmac-sha384-192`) in environments where weaker alternatives are not allowed. This lack of support makes the _SunJGSS_ security provider unable to do Kerberos authentication with _SunPKCS11_ in such environments.

Find more information about the PKCS #11 mechanism for AES CBC-CTS in the following sections of the standard:

* Section [2.10 AES CBC with Cipher Text Stealing CTS](https://docs.oasis-open.org/pkcs11/pkcs11-curr/v2.40/errata01/pkcs11-curr-v2.40-errata01-complete.html#_Toc441850501) of v2.40
* Section [2.12 AES CBC with Cipher Text Stealing CTS](https://docs.oasis-open.org/pkcs11/pkcs11-curr/v3.0/pkcs11-curr-v3.0.html#_Toc30061253) of v3.0

The PKCS #11 definition of `CKM_AES_CTS` refers to the [Addendum to NIST Special Publication 800-38A, "Recommendation for Block Cipher Modes of Operation: Three Variants of Ciphertext Stealing for CBC Mode"](https://doi.org/10.6028/NIST.SP.800-38A-Add) document but does not prescribe which of the CTS variants to use. This left the door open for PKCS #11 implementers to make different assumptions and create interoperability issues. As an example, the NSS Software Token implemented the _CS1_ variant (see [here](https://github.com/nss-dev/nss/blob/NSS_3_90_RTM/lib/freebl/cts.c#L64-L65)) but public clarification had to be given [here](https://bugzilla.mozilla.org/show_bug.cgi?id=373108#c7). For OpenJDK to support different PKCS #11 tokens, a _SunPKCS11_ configuration parameter to specify the token's CTS variant is required.

The _SunJCE_ security provider uses the CTS variant _CS3_ (see [here](https://github.com/openjdk/jdk/blob/jdk-22-ga/src/java.base/share/classes/com/sun/crypto/provider/CipherTextStealing.java#L32-L37)), expected for Kerberos authentication. To allow interoperability with _SunJCE_ and Kerberos, _SunPKCS11_ should convert the variant used by the token to _CS3_ and vice versa.

Solution
--------

Enhance the _SunPKCS11_ security provider to support the following AES CBC-CTS transformations for the `Cipher` service type:

* `AES/CTS/NoPadding`
* `AES_128/CTS/NoPadding`
* `AES_192/CTS/NoPadding`
* `AES_256/CTS/NoPadding`

Introduce a new _SunPKCS11_ configuration parameter named `cipherTextStealingVariant` that can be optionally set with any of the following values: `CS1` (default when the parameter is not set), `CS2` and `CS3`. This parameter allows specifying the token's CTS variant. After encryption, the ciphertext will be converted from the token's variant to _CS3_. Before decryption, the ciphertext will be converted from _CS3_ to the token's variant.

Specification
-------------

Make the following changes to the [PKCS#11 Reference Guide](https://docs.oracle.com/en/java/javase/22/security/pkcs11-reference-guide1.html).

### Update [Table 5-1 Attributes in the PKCS#11 Provider Configuration File](https://docs.oracle.com/en/java/javase/22/security/pkcs11-reference-guide1.html#GUID-C4ABFACB-B2C9-4E71-A313-79F881488BB9__GUID-8ADD17FD-3320-4D8F-A1ED-319DFEABA338) to add the following row:

| Attribute                   | Value                                                   | Description |
|:----------------------------|:--------------------------------------------------------|:------------|
| `cipherTextStealingVariant` | Possible values: `CS1`, `CS2`, or `CS3`; default: `CS1` | CTS mode variant used by the token, as described in [Addendum to NIST Special Publication 800-38A, "Recommendation for Block Cipher Modes of Operation: Three Variants of Ciphertext Stealing for CBC Mode"](https://doi.org/10.6028/NIST.SP.800-38A-Add). The PKCS#11 standard does not prescribe which variant to use for [CKM_AES_CTS](https://docs.oasis-open.org/pkcs11/pkcs11-curr/v3.0/pkcs11-curr-v3.0.html#_Toc30061253) and implementers may make different choices. After encryption, the ciphertext will be converted from the token's variant to _CS3_. Before decryption, the ciphertext will be converted from _CS3_ to the token's variant. |
